Scheme-langserver

คุณสามารถอ่าน เปเปอร์ของฉัน และอ้างอิงได้ดังนี้

WANG, Z. (2025, May 12). Scheme-langserver: Treat Scheme Code Editing as the First-Class Concern. The 18th European Lisp Symposium (ELS`25), Zurich. https://doi.org/10.5281/zenodo.15384882

เนื่องจาก GitHub อาจบล็อกที่อยู่ IP ของจีน ฉันจึงต้องคัดลอกที่เก็บนี้ไปไว้ที่ ที่นี่ และ ที่นี่ ฉันจะอัปเดตที่เก็บทั้งหมดอย่างต่อเนื่องแต่ไม่สามารถรับประกันได้ ขณะนี้ฉันยังร่วมมือกับ XmacsLabs คุณอาจพบ fork ได้ ที่นี่

ขณะนี้ VSCode สามารถใช้ scheme-langserver ได้แล้ว! โปรดดู ที่นี่

หมายเหตุ: มีบั๊กจำนวนมากใน scheme-langserver ฉันกำลังแก้ไขและขอความช่วยเหลือจากชุมชน โปรดอดทน

หมายเหตุ: คุณสามารถดูข้อมูลประเภทที่สร้างอัตโนมัติ ที่นี่ ตอนนี้ใช้หลักสำหรับการพัฒนาขั้นถัดไป (อาจรวมถึง AKKU) และการดีบัก

การพัฒนาฟีเจอร์ต่าง ๆ เช่น autocomplete, goto definition หรือแสดงเอกสารขณะวางเมาส์ถือเป็นงานใหญ่สำหรับการเขียนโปรแกรม อย่างไรก็ตาม เมื่อเทียบกับภาษาอื่น ๆ เช่น java, python, javascript และ c การพัฒนา language server protocol สำหรับภาษา lisp นั้นเหมือนทำอยู่ในสุญญากาศ Geiser, racket langserver และ swish-lint ฯลฯ งานของพวกเขาทั้งหมดอาศัย repl(Read-Eval-Print Loop) หรือ keyword tokenizer แทนที่จะเป็นการเขียนโปรแกรมจริง เช่น ถ้านักพัฒนากำลังเขียนโค้ดในโปรเจกต์ที่ยังไม่เสร็จสมบูรณ์ ซึ่งโค้ดยังไม่สามารถรันได้ทั้งหมด Geiser หรือเครื่องมืออื่น ๆ จะทำได้แค่เติม identifier ที่ binding ไว้ในระดับบนสุดซึ่งแสดงโดย procedure environment-symbols (สำหรับ Chez) หรือเติม symbol แทน identifier ซึ่งหมายความว่า สำหรับ local binding และโค้ดที่ยังไม่สมบูรณ์ แม้ว่าการสนับสนุนด้าน programming จะมีความสำคัญมากที่สุด Geiser และเครื่องมืออื่น ๆ ก็ไม่ช่วยอะไรในการระบุขอบเขตที่ถูกต้องของ identifier กรณีเดียวกันนี้เกิดขึ้นกับ goto definition และฟีเจอร์อื่น ๆ อีกมากมาย

สาเหตุหลักคือ สำหรับ scheme และ lisp dialects อื่น ๆ ชุดข้อมูลที่หลากหลายและโครงสร้างควบคุมที่ยืดหยุ่นทำให้การวิเคราะห์โปรแกรมเป็นความท้าทายครั้งใหญ่ โดยเฉพาะ macro ดูเหมือนว่า scheme จะใช้หลัก ๆ เพื่อ meta/macro programming สำหรับคนเก่งเท่านั้น แต่ฉันไม่เห็นด้วย Scheme สามารถสร้างสิ่งที่น่าสนใจมากมายหากมีสภาพแวดล้อมการเขียนโปรแกรมที่ดี และตอนนี้คุณสามารถ เริ่มต้นได้ที่นี่

แพ็กเกจนี้เป็นการพัฒนา language server protocol ที่ช่วยในการเขียนโปรแกรม scheme โดยให้ฟีเจอร์ completion, definition และ type inference ฟีเจอร์เหล่านี้สร้างขึ้นบนการวิเคราะห์โค้ดแบบ static ตาม r6rs standard และกฎที่ชัดเจนสำหรับโค้ดที่ยังไม่สมบูรณ์ แพ็กเกจนี้และไลบรารีที่เกี่ยวข้องจะถูกเผยแพร่หรือกำลังจะเผยแพร่ใน Akku ซึ่งเป็น package manager สำหรับ Scheme

แพ็กเกจนี้ยังผ่านการทดสอบกับ Chez Scheme เวอร์ชัน 9.4 และ 9.5 โดยจะทดสอบอย่างละเอียดกับเวอร์ชัน 10.0.0 หลังจากอัปเกรด OS ของแล็ปท็อปเป็นเวอร์ชันใหม่

ฉันทำ open source นี้ในเวลาว่างเท่านั้น และฉันสามารถนำไอเดียดี ๆ มาสู่ชุมชนได้มากมาย เช่น การฝัง data flow analysis ลงใน scheme-langserver หรือสิ่งอื่น ๆ อีกมากมาย และฉันกำลังขอรับบริจาคหรือทุนสนับสนุนอย่างต่อเนื่อง คุณสามารถบริจาคผ่าน patreon หน้านี้ หรือ 爱发电

local util = require 'lspconfig.util'

return {
  default_config = {

    filetypes = { 'scheme' },
    root_dir = util.find_git_ancestor,
    single_file_support = true,
  },
  docs = {

การแก้ไขซ้ำด้วย Log

ด้วย output log ข้างต้น คุณสามารถใช้ tests/log-debug.sps เพื่อตรวจสอบบั๊กที่เกิดซ้ำ:

  1. เปลี่ยนชื่อ {path-to-log} (โดยปกติคือ ~/scheme-langserver.log) เป็น ~/ready-for-analyse.log;
  2. รัน scheme --script tests/log-debug.sps หากคุณต้องการจำลองสภาพแวดล้อมแบบมัลติเธรด ก็สามารถรัน scheme --script tests/log-debug.sps ได้เช่นกัน
